## 1. Does Cloudflare Pages Automatically Noindex the Production `.pages.dev` URL?

**Answer: NO. Automatic noindex only applies to preview deployments, NOT the production `.pages.dev` subdomain.**

From the official Cloudflare Pages documentation:

> "By default, every preview deployment generated by Cloudflare Pages includes the
> `X-Robots-Tag: noindex` HTTP response header, which acts as a directive to search engine
> crawlers, instructing them to disregard the page."

The key distinction:

| URL Type                                                            | Automatic Noindex?                                          |
| ------------------------------------------------------------------- | ----------------------------------------------------------- |
| `*.pages.dev` **preview deployments** (branch deploys, PR previews) | YES — Cloudflare adds `X-Robots-Tag: noindex` automatically |
| `yourproject.pages.dev` **production deployment** (main branch)     | NO — no automatic noindex                                   |
| Custom domain (`alexmayhew.dev`)                                    | NO — never noindexed by Cloudflare                          |

**Implication for alexmayhew.dev:** The production URL `alexmayhew-dev.pages.dev` is publicly accessible, crawlable by Google, and NOT automatically protected. It serves identical content to `alexmayhew.dev`. This creates a legitimate duplicate content risk.

**Severity of the risk:** Google is sophisticated enough to usually pick the custom domain as the canonical version due to it having more backlinks and authority. However, it is not guaranteed, and having two identical URLs without explicit signals wastes crawl budget and can confuse canonical selection. The `.pages.dev` URL has been observed appearing in Google search results for sites that did not explicitly block it.

## 2. The Four Available Approaches (with definitive verdict on each)

### Approach A: `X-Robots-Tag: noindex` in Custom Worker (RECOMMENDED)

**The correct solution for this stack.** The custom-worker.ts already exists at the project root and wraps all HTML responses. Adding a host-conditional `X-Robots-Tag` header here is one line of code and is guaranteed to fire for every request to the `.pages.dev` URL.

**Why this is the definitive approach for OpenNext:**

- `public/_headers` does NOT apply to Worker-generated HTML responses (documented Cloudflare limitation). It only applies to static CDN assets. Adding X-Robots-Tag to `_headers` would cover static files but NOT HTML pages served by the Worker.
- Next.js middleware has the same unreliability problem on OpenNext/Cloudflare — middleware headers can be stripped before reaching the browser (see `opennext-middleware-headers-cloudflare-2026.md`).
- The custom worker runs for every request before any response reaches the client. It is the only layer that is guaranteed to execute on all HTML responses regardless of cache state.

**Implementation — extend the existing `custom-worker.ts`:**

```typescript
async fetch(request: Request, env: CloudflareEnv, ctx: ExecutionContext): Promise<Response> {
  const response = await handler.fetch(request, env, ctx);

  const contentType = response.headers.get("content-type") ?? "";
  if (!contentType.includes("text/html")) {
    return response;
  }

  const newHeaders = new Headers(response.headers);

  // Prevent .pages.dev production URL from being indexed — it duplicates alexmayhew.dev.
  // Preview deployments already get noindex from Cloudflare automatically.
  const host = request.headers.get("host") ?? "";
  if (host.endsWith(".pages.dev")) {
    newHeaders.set("X-Robots-Tag", "noindex, nofollow");
  }

  // Inject security headers (existing logic)
  for (const [name, value] of Object.entries(SECURITY_HEADERS)) {
    newHeaders.set(name, value);
  }

  return new Response(response.body, {
    status: response.status,
    statusText: response.statusText,
    headers: newHeaders,
  });
},
```

**What this achieves:**

- Every HTML response served from `alexmayhew-dev.pages.dev` gets `X-Robots-Tag: noindex, nofollow`
- Requests from `alexmayhew.dev` (the custom domain) are unaffected — host check does not match
- Preview deployments remain unaffected (Cloudflare already handles them)
- Zero Cloudflare dashboard changes required — fully version-controlled in code

---

### Approach B: Cloudflare Dashboard Redirect Rules (CANNOT target `.pages.dev`)

**This approach does not work.** Transform Rules, Redirect Rules, Page Rules, and WAF Custom Rules all operate on **zones** you control in your Cloudflare account. `pages.dev` is a Cloudflare-managed domain that is NOT a zone in your account.

From the community confirmation:

> "Pages.dev is not a zone in your account, so your rules will not apply there."

The only exception: if you proxy requests to `.pages.dev` through your own zone via a custom domain or Worker route — but that defeats the purpose (you'd already be controlling it at code level).

**Verdict: Not available. Dashboard rules cannot target `*.pages.dev`.**

---

### Approach C: `_headers` File Entry for `.pages.dev`

Adding `X-Robots-Tag: noindex` to `public/_headers` under `https://yourproject.pages.dev/*` does partially work — **for static assets** (JS, CSS, images, fonts served by the Cloudflare CDN layer). However:

- HTML pages are NOT static assets — they are Worker-generated responses
- `_headers` is explicitly documented to NOT apply to Worker/Pages Function responses
- You would protect the asset URLs (which crawlers generally don't index anyway) but leave HTML pages unprotected

This is what multiple community posts suggest, but it is incomplete for SSR/OpenNext deployments.

**Verdict: Insufficient for HTML pages. Covers CDN-served static assets only.**

---

### Approach D: 301 Redirect from `.pages.dev` to Custom Domain

**Technically not possible to implement at the Cloudflare platform level** for the `.pages.dev` domain (same zone restriction as Approach B). You cannot configure a Bulk Redirect rule that targets `alexmayhew-dev.pages.dev` because it is not a zone you manage.

However, you CAN implement this **inside the Worker itself** — a 301 redirect from the Worker's fetch handler before any content is served:

```typescript
async fetch(request: Request, env: CloudflareEnv, ctx: ExecutionContext): Promise<Response> {
  const host = request.headers.get("host") ?? "";

  // Hard redirect .pages.dev to the canonical custom domain
  if (host.endsWith(".pages.dev")) {
    const url = new URL(request.url);
    url.hostname = "alexmayhew.dev";
    return Response.redirect(url.toString(), 301);
  }

  // ... rest of handler
}
```

**Tradeoffs of redirect vs. noindex:**

| Factor                                       | 301 Redirect                                           | X-Robots-Tag: noindex                   |
| -------------------------------------------- | ------------------------------------------------------ | --------------------------------------- |
| SEO signal strength                          | Strongest (Google treats redirect as canonical signal) | Strong (X-Robots-Tag is definitive)     |
| Latency impact                               | Adds one round-trip for every `.pages.dev` request     | None                                    |
| Breaks direct `.pages.dev` access            | Yes — all `.pages.dev` traffic redirects               | No — page still loads, just not indexed |
| Impact on Cloudflare Pages dashboard preview | Depends on host matching logic                         | None                                    |
| Complexity                                   | Slightly higher                                        | Minimal                                 |
| SEO outcome                                  | Identical for final result                             | Identical for final result              |

**Verdict on redirect approach:** It is a valid alternative but has no SEO advantage over noindex for this use case. The `.pages.dev` URL is not receiving meaningful external links that need to be redirected for link equity. Noindex is simpler, does not break direct URL access (useful for internal QA), and is equally effective at preventing duplicate indexing.

**The consensus from SEO best practices:** Use 301 redirect when you are _retiring_ a URL that may have accumulated backlinks or traffic. Use noindex when you want to prevent indexing without breaking the URL for legitimate non-crawler access. For a `.pages.dev` staging/secondary URL, noindex is the correct choice.

## 3. Should You Also Add Canonical URLs Pointing to the Custom Domain?

**Yes, as a belt-and-suspenders measure, but canonicals alone are insufficient for this specific case.**

The existing research (`nextjs-canonical-url-best-practices-2026.md`) establishes that every page should have an explicit self-referencing canonical. For pages served from `alexmayhew-dev.pages.dev`, those canonical tags already point to `https://alexmayhew.dev/...` (because the canonical is set statically in Next.js metadata using `metadataBase: new URL('https://alexmayhew.dev')`).

**Why canonicals alone are not enough:**

1. Google treats canonical as a "hint" not a directive — it can and does override canonicals when it has conflicting signals
2. A `.pages.dev` URL with canonical pointing to the custom domain will often work correctly, but Google may still crawl and process the `.pages.dev` version, consuming crawl budget
3. `X-Robots-Tag: noindex` is a directive (not a hint) — Google must honor it
4. Having both signals (canonical pointing to custom domain + noindex on `.pages.dev`) is the defense-in-depth approach

**Verdict:** The canonicals are already correct (pointing to `alexmayhew.dev`) due to the static `metadataBase`. The `X-Robots-Tag: noindex` in the worker adds the definitive crawl directive on top. Both should be in place.

## 4. Cloudflare-Specific Edge Solutions (Dashboard Rules, WAF, Transform Rules)

As established in Approach B: **none of the Cloudflare dashboard tools can target `*.pages.dev`** because that domain is not a managed zone in your account.

The only Cloudflare-specific solutions that work are:

1. **`_headers` file** — covers static CDN assets only, incomplete for HTML
2. **Custom Worker code** — the only complete solution (Approach A above)

There is no dashboard-only solution. Code changes are required.

## 5. Where in OpenNext to Inject the Header

**The definitive answer: in `custom-worker.ts`, not in Next.js middleware.**

Reasons:

| Layer                            | Works for `.pages.dev` Noindex?      | Reliability            |
| -------------------------------- | ------------------------------------ | ---------------------- |
| `public/_headers`                | No — HTML pages excluded             | Static assets only     |
| `next.config.ts` `headers()`     | Partially — may miss some routes     | Unreliable on OpenNext |
| Next.js `middleware.ts`          | Partially — headers may be stripped  | Unreliable on OpenNext |
| **`custom-worker.ts` (CORRECT)** | **Yes — guaranteed for all HTML**    | **100% reliable**      |
| Cloudflare dashboard rules       | No — `.pages.dev` not a managed zone | N/A                    |

The `custom-worker.ts` sits at the outermost layer of the request pipeline. It receives every request and can inspect and modify every response before it leaves Cloudflare. The host check (`request.headers.get("host")`) is always accurate because this is the actual request host, not a forwarded or rewritten value.

## 6. Recommended Implementation for alexmayhew.dev

The `custom-worker.ts` already exists and already injects security headers. The change is minimal — add the host check before the security headers loop:

```typescript
async fetch(request: Request, env: CloudflareEnv, ctx: ExecutionContext): Promise<Response> {
  const response = await handler.fetch(request, env, ctx);

  const contentType = response.headers.get("content-type") ?? "";
  if (!contentType.includes("text/html")) {
    return response;
  }

  const newHeaders = new Headers(response.headers);

  // Noindex the .pages.dev production URL.
  // Cloudflare only auto-noindexes preview deployments, not the production .pages.dev.
  // public/_headers cannot cover HTML pages (Worker responses), so this is the only
  // reliable injection point.
  const host = request.headers.get("host") ?? "";
  if (host.endsWith(".pages.dev")) {
    newHeaders.set("X-Robots-Tag", "noindex, nofollow");
  }

  // Security headers (existing)
  for (const [name, value] of Object.entries(SECURITY_HEADERS)) {
    newHeaders.set(name, value);
  }

  return new Response(response.body, {
    status: response.status,
    statusText: response.statusText,
    headers: newHeaders,
  });
},
```

**Verification after deploy:**

```bash
# Should show X-Robots-Tag: noindex, nofollow
curl -sI https://alexmayhew-dev.pages.dev/ | grep -i x-robots

# Should NOT show X-Robots-Tag (or show nothing matching noindex)
curl -sI https://alexmayhew.dev/ | grep -i x-robots
```

## 7. Complete Defense-in-Depth Strategy (All Layers)

| Layer                                   | What it does                                                                                | Status                                    |
| --------------------------------------- | ------------------------------------------------------------------------------------------- | ----------------------------------------- |
| `custom-worker.ts` host check           | `X-Robots-Tag: noindex, nofollow` on all `.pages.dev` HTML                                  | **ADD THIS**                              |
| Next.js `metadata.alternates.canonical` | Points to `alexmayhew.dev` on every page                                                    | Already correct via `metadataBase`        |
| Cloudflare Preview Deployments          | Automatic `X-Robots-Tag: noindex` on branch/PR previews                                     | Already in place (Cloudflare platform)    |
| `public/_headers`                       | `X-Robots-Tag` for static assets on `.pages.dev` (optional belt-and-suspenders)             | Can add but minimal additional protection |
| robots.txt                              | Does NOT block `.pages.dev` (you don't control the zone for a robots.txt to be served from) | N/A                                       |

The worker-level noindex + correct canonical is the complete and sufficient solution.

## 8. Why NOT Use Next.js Middleware for This

The tempting approach is to add the host check in `src/middleware.ts` since that is where request-level logic typically lives in Next.js. This does not work reliably for the following documented reasons:

1. **Middleware headers can be stripped by OpenNext's internal pipeline** before they reach the browser. The existing research documents this as the root cause of security headers missing in production.
2. **R2 ISR cache hits bypass middleware entirely** when cache interception is enabled.
3. **`public/_headers` overrides**: Since `_headers` is processed at the Cloudflare CDN layer before the Worker runs, it applies to static assets regardless of what middleware sets.

The custom worker wrapper executes AFTER the OpenNext handler returns a response, at the outermost layer. There is no pipeline step between the worker and the browser that can strip these headers.

## Key Findings Summary

1. **Cloudflare does NOT automatically noindex the production `.pages.dev` URL.** Only preview (branch/PR) deployment URLs get automatic `X-Robots-Tag: noindex`. The production `yourproject.pages.dev` URL is fully crawlable.

2. **Dashboard rules (Transform Rules, WAF, Page Rules) cannot target `.pages.dev`** — it is not a zone in your account. Code-level implementation is required.

3. **The correct solution for OpenNext is the custom worker wrapper** (`custom-worker.ts`). Add a host check before the existing security headers loop. One conditional, one `newHeaders.set()` call.

4. **`public/_headers` is insufficient** for HTML pages on this stack. It covers CDN-served static assets only, not Worker-generated HTML responses.

5. **Next.js middleware is unreliable** for this use case due to known OpenNext header stripping behavior.

6. **301 redirects from `.pages.dev` to the custom domain** are technically possible in the worker but offer no SEO advantage over noindex for this use case, and break direct URL access for QA.

7. **Canonical URLs alone are insufficient** — they are hints, not directives. The `X-Robots-Tag: noindex` directive is required alongside the correct canonical.

8. **Both signals are already partially in place**: canonical points to custom domain (via `metadataBase`). The missing piece is `X-Robots-Tag: noindex` on `.pages.dev` HTML responses.

# OpenNext Cloudflare: Middleware Response Headers Not Appearing in Production (2026-03-14)

**Status:** CURRENT
**Session:** Root cause analysis for CSP, X-Frame-Options, Referrer-Policy, Permissions-Policy missing in production on alexmayhew.dev

---

## TL;DR

There are **three independent causes** working together to suppress middleware-set headers in production. The most critical is that the `public/_headers` file (which already exists) only applies to static assets — not Worker-generated HTML responses. The second is that the `missing` matcher condition is correct in intent but only a performance optimization for prefetches — it does NOT prevent headers from appearing on normal requests. The third is that R2 ISR cache serving may bypass middleware entirely when cache interception is enabled.

**The correct fix is a custom worker wrapper** that applies security headers to all responses at the Worker fetch handler level, since that is the only layer that executes for every request regardless of cache state or asset routing.

---

## The Three Root Causes

### Cause 1: `public/_headers` does NOT apply to Worker-generated responses

The existing `public/_headers` file (line 1-44) sets `X-Frame-Options`, `X-Content-Type-Options`, `Referrer-Policy`, `Permissions-Policy`, and `X-XSS-Protection` under `/*`.

**This file only applies to static assets served directly by the Cloudflare Assets layer (CDN), NOT to HTML pages generated by the Worker.**

From official Cloudflare Workers documentation (`developers.cloudflare.com/workers/static-assets/headers/`):

> "Custom headers defined in the `_headers` file are not applied to responses generated by your Worker code, even if the request URL matches a rule defined in `_headers`."

For the Cloudflare Pages `_headers` docs, same restriction:

> "Custom headers defined in the `_headers` file are not applied to responses generated by Pages Functions."

**What this means for alexmayhew.dev:**

- `/_next/static/*` assets — `Cache-Control: public, max-age=31536000, immutable` → WORKS (static asset)
- `/images/*`, `/*.webp`, `/*.svg` — cache headers → WORKS (static asset)
- `/*` HTML pages — X-Frame-Options, Referrer-Policy, etc. → DOES NOT WORK (Worker response)

HSTS and `x-content-type-options` appearing in production is almost certainly from Cloudflare's built-in security settings ("Security Headers" in SSL/TLS → Edge Certificates dashboard), not from middleware or `_headers`.

---

### Cause 2: The `missing` matcher condition is a performance hint, not a correctness issue

The current `middleware.ts` matcher:

```typescript
export const config = {
	matcher: [
		{
			source: "/((?!api|_next/static|_next/image|favicon.ico).*)",
			missing: [
				{ type: "header", key: "next-router-prefetch" },
				{ type: "header", key: "purpose", value: "prefetch" },
			],
		},
	],
};
```

**The `missing` condition is correctly designed.** It means: "only run middleware when the request does NOT have `next-router-prefetch` OR `purpose: prefetch` headers." This correctly skips middleware for Next.js client-side prefetch requests, which don't need security headers because they're not full page navigations.

For all regular browser navigations (page loads), neither `next-router-prefetch` nor `purpose: prefetch` will be present, so middleware WILL run. This condition is not causing the missing headers.

There is an open Next.js issue (#63728) that `next-router-prefetch` is sometimes absent from prefetch requests in middleware, making this filter a no-op for prefetches — but this does NOT affect normal requests. Normal requests always satisfy the `missing` condition and middleware runs for them.

**Verdict: The matcher is not the problem.**

---

### Cause 3: R2 ISR cache + `run_worker_first` can bypass middleware for HTML pages

OpenNext Cloudflare has two asset serving modes:

**`run_worker_first: false` (default for static assets):**

- Static files in `.open-next/assets` are intercepted by Cloudflare's CDN layer BEFORE reaching the Worker
- The Worker (and thus middleware) never runs for these requests
- This applies to `_next/static/*`, `public/` files, images, etc.
- These are NOT billed as Worker invocations

**`run_worker_first: true`:**

- All requests reach the Worker first
- Middleware runs for all requests
- More expensive but headers from Worker code apply everywhere

For **dynamically-rendered HTML pages** (SSR routes), the Worker always runs. Middleware should execute.

However, there is a subtle interaction with **R2 incremental cache**:

When `enableCacheInterception: true` is set in `open-next.config.ts`, cached ISR/SSG pages can be served directly from R2 without invoking the full Next.js server. The OpenNext docs note: "when a page can be retrieved from the cache, the main server can be fully bypassed." Whether middleware headers are applied to these cached responses is not documented but may depend on where in the pipeline the interception occurs.

The current `open-next.config.ts` does NOT enable cache interception:

```typescript
export default defineCloudflareConfig({
	incrementalCache: r2IncrementalCache,
	// enableCacheInterception: not set — defaults to false
});
```

This means cache interception is NOT currently active, so this cause is probably not the immediate problem. However, it is worth noting for the future.

**Verdict: Cause 3 is NOT currently active, but Cause 1 is the primary issue.**

## Is Middleware Running At All?

Yes, middleware is running in production — with caveats.

**Evidence middleware IS running:**

- The `missing` filter is correctly designed and regular browser requests satisfy it
- `NextResponse.next()` is used, which should propagate headers
- The wrangler config has `nodejs_compat` and `compatibility_date: 2025-12-01`, both of which support middleware execution

**The most likely explanation:** Middleware runs and sets headers on the `NextResponse` object, but those headers are being overridden or stripped at the Cloudflare edge before reaching the browser.

Specifically, Cloudflare's "Security Headers" feature in the dashboard (`SSL/TLS → Edge Certificates → HTTP Strict Transport Security`) automatically adds HSTS. Some Cloudflare plans also inject `X-Content-Type-Options: nosniff`. These Cloudflare-level headers override Worker response headers for the same header names, which explains why only HSTS and `x-content-type-options` appear — they're from Cloudflare's own injection, not middleware.

For CSP, X-Frame-Options (the more permissive form), Referrer-Policy, and Permissions-Policy, Cloudflare does NOT inject them automatically. If middleware is running and setting them on `NextResponse`, they should appear. The fact that they don't suggests middleware headers are being stripped somewhere in the OpenNext → Cloudflare response pipeline.

## Is There a Known OpenNext Bug for This?

**Not a documented first-class bug for security response headers specifically.**

Related GitHub issues in `opennextjs/opennextjs-cloudflare`:

| Issue | What it covers                                                               | Status                                                                            |
| ----- | ---------------------------------------------------------------------------- | --------------------------------------------------------------------------------- |
| #606  | Set-Cookie header ORDER differs from Vercel (middleware header precedence)   | Open                                                                              |
| #501  | Multiple Set-Cookie headers folded (merged into one) in middleware responses | Open                                                                              |
| #585  | CORS headers (`access-control-allow-origin`) missing from static assets      | Closed — documented limitation, use `_headers` for assets or route through Worker |
| #809  | Location redirect header uses absolute URL instead of relative               | Open                                                                              |

Issue #585's resolution is directly relevant: the maintainers confirmed "headers are not applied to static assets at the moment" and pointed to `_headers` for static assets and Worker code for dynamic responses.

There is no documented bug specifically about security headers (CSP, X-Frame-Options, etc.) being stripped from Worker responses. This reinforces the hypothesis that the `_headers` file is being relied upon for HTML pages when it only works for static assets.

## What Are the Correct Solutions?

Three viable approaches exist. They are not mutually exclusive.

---

### Solution A: Custom Worker Wrapper (RECOMMENDED — most reliable)

Create a `custom-worker.ts` that wraps the generated OpenNext worker and appends security headers to every response.

This is the ONLY approach that guarantees headers on ALL responses regardless of:

- Whether the page is statically generated or SSR
- Whether R2 cache interception is enabled in the future
- Whether Cloudflare's CDN layer serves the response or the Worker does

**File: `custom-worker.ts`** (project root, alongside `wrangler.jsonc`):

```typescript
// @ts-ignore — .open-next/worker.js is generated at build time
import { default as handler } from "./.open-next/worker.js";

// Re-export DO handlers if using Durable Objects for cache
// @ts-ignore
export { DOQueueHandler, DOShardedTagCache } from "./.open-next/worker.js";

const SECURITY_HEADERS: Record<string, string> = {
	"Content-Security-Policy": [
		"default-src 'self'",
		"script-src 'self' 'unsafe-inline' https://static.cloudflareinsights.com https://challenges.cloudflare.com https://*.googletagmanager.com",
		"style-src 'self' 'unsafe-inline'",
		"img-src 'self' blob: data: https://*.google-analytics.com https://*.googletagmanager.com https://*.g.doubleclick.net https://*.google.com",
		"font-src 'self'",
		"connect-src 'self' https://cloudflareinsights.com https://challenges.cloudflare.com https://*.ingest.sentry.io https://*.google-analytics.com https://*.analytics.google.com https://*.googletagmanager.com https://*.g.doubleclick.net https://*.google.com",
		"frame-src 'self' https://challenges.cloudflare.com https://www.googletagmanager.com",
		"object-src 'none'",
		"base-uri 'self'",
		"form-action 'self'",
		"frame-ancestors 'none'",
		"upgrade-insecure-requests",
	].join("; "),
	"X-Frame-Options": "DENY",
	"X-Content-Type-Options": "nosniff",
	"Referrer-Policy": "strict-origin-when-cross-origin",
	"Permissions-Policy": "camera=(), microphone=(), geolocation=(), browsing-topics=()",
};

export default {
	async fetch(request: Request, env: CloudflareEnv, ctx: ExecutionContext): Promise<Response> {
		const response = await handler.fetch(request, env, ctx);

		// Only inject security headers on HTML responses.
		// Static assets (JS, CSS, images) don't need CSP and X-Frame-Options.
		const contentType = response.headers.get("content-type") ?? "";
		if (!contentType.includes("text/html")) {
			return response;
		}

		const newHeaders = new Headers(response.headers);
		for (const [name, value] of Object.entries(SECURITY_HEADERS)) {
			// Don't overwrite if already set (e.g., from middleware or next.config)
			if (!newHeaders.has(name)) {
				newHeaders.set(name, value);
			}
		}

		return new Response(response.body, {
			status: response.status,
			statusText: response.statusText,
			headers: newHeaders,
		});
	},
} satisfies ExportedHandler<CloudflareEnv>;
```

**Update `wrangler.jsonc` to point to the custom worker:**

```jsonc
{
	"main": "./custom-worker.ts", // Changed from ".open-next/worker.js"
	// ... rest of config unchanged
}
```

**Caveats:**

- If `middleware.ts` also sets the same headers, they appear in the `response.headers` passed to the custom worker — the `if (!newHeaders.has(name))` guard avoids doubling. Alternatively, use `newHeaders.set(name, value)` unconditionally to ensure the custom worker is the source of truth.
- The `middleware.ts` can be kept or simplified — since the custom worker is the guaranteed layer, middleware's header-setting becomes redundant for security headers.
- CSP in the custom worker is static. Nonce-based CSP cannot be done here because nonces must be generated per-request and injected into HTML, which requires middleware + layout plumbing.

### Solution B: Keep Middleware, Fix the Missing Headers via `next.config.ts` Headers API

The `headers()` function in `next.config.ts` runs at the Next.js routing layer and should apply to all pages (before filesystem routes). For OpenNext, this runs inside the Worker.

```typescript
// next.config.ts
const nextConfig: NextConfig = {
	async headers() {
		return [
			{
				source: "/((?!_next/static|_next/image|favicon.ico).*)",
				headers: [
					{ key: "X-Frame-Options", value: "DENY" },
					{ key: "X-Content-Type-Options", value: "nosniff" },
					{ key: "Referrer-Policy", value: "strict-origin-when-cross-origin" },
					{
						key: "Permissions-Policy",
						value: "camera=(), microphone=(), geolocation=(), browsing-topics=()",
					},
					// CSP here as a static string — no nonce support
					{ key: "Content-Security-Policy", value: "default-src 'self'; ..." },
				],
			},
		];
	},
};
```

**Limitation for OpenNext Cloudflare:** From official OpenNext docs: "the `headers` option of `next.config.ts` doesn't apply to `public` files and immutable build files (like `_next/static`)." For HTML pages served through the Worker, this SHOULD work — but community reports suggest it may not be reliable on all OpenNext versions.

**Verdict:** Less reliable than the custom worker approach. Use as a belt-and-suspenders addition but not as the sole mechanism.

---

### Solution C: Cloudflare Dashboard Transform Rules

Cloudflare's HTTP Response Header Modification rules (under Rules → Transform Rules in the dashboard) can add static headers to all responses at the Cloudflare edge.

**What they can do:**

- Add static CSP, X-Frame-Options, Referrer-Policy, Permissions-Policy to all page responses
- Apply before the browser sees the response
- Work regardless of whether the response came from the Worker or the CDN cache

**What they cannot do:**

- Generate per-request nonces (no dynamic values)
- Read or modify response body
- Differ by content type (limited conditional logic on the free plan)

**How to configure:** Cloudflare Dashboard → your domain → Rules → Transform Rules → HTTP Response Headers → Create Rule → "Add" action for each header.

**Verdict:** A reliable fallback but harder to version-control and audit. Use if the custom worker approach introduces complexity. Not suitable for nonce-based CSP. Good for organizations preferring network-layer security headers independent of code deployments.

## Why HSTS and X-Content-Type-Options Appear But Others Don't

**HSTS** (`Strict-Transport-Security`) is injected by Cloudflare's HSTS feature under SSL/TLS → Edge Certificates → HTTP Strict Transport Security. It is a Cloudflare platform feature and has nothing to do with middleware.

**`X-Content-Type-Options: nosniff`** is injected by Cloudflare automatically for zone security settings on Pro and above plans (or via Security Headers in the dashboard). It is also a Cloudflare platform feature.

Neither comes from `middleware.ts` — they're from the Cloudflare platform layer.

The headers middleware is attempting to set (CSP, X-Frame-Options, Referrer-Policy, Permissions-Policy) are NOT injected by Cloudflare automatically. If middleware was reliably setting them, they would appear. The fact they don't confirms the `public/_headers` confusion as the root cause: the `_headers` file was trusted to handle `/*` but it doesn't apply to Worker-generated HTML.

## Decision Matrix for This Project

| Approach                         | Reliable for HTML             | Works with Nonces    | Version-Controlled  | Effort                    |
| -------------------------------- | ----------------------------- | -------------------- | ------------------- | ------------------------- |
| `public/_headers`                | NO — static assets only       | No                   | Yes                 | Already done              |
| `middleware.ts` response headers | PARTIALLY — may be stripped   | Yes                  | Yes                 | Already done, not working |
| Custom worker wrapper            | YES — guaranteed              | No (static CSP only) | Yes                 | Low                       |
| `next.config.ts` headers()       | MOSTLY — may miss some routes | No                   | Yes                 | Low                       |
| Cloudflare Transform Rules       | YES                           | No                   | No (dashboard-only) | Low                       |

**Recommended approach for alexmayhew.dev:**

1. **Custom worker wrapper** as the guaranteed layer for CSP, X-Frame-Options, Referrer-Policy, Permissions-Policy on HTML responses
2. Keep `middleware.ts` as-is but understand it may be redundant for headers — it correctly handles the nonce plumbing path if nonce-based CSP is implemented later
3. Keep `public/_headers` for cache headers on static assets (`_next/static/*`, images, fonts) — it works correctly for those
4. Cloudflare dashboard HSTS and X-Content-Type-Options can remain in platform settings (no action needed)

## Impact on Current `public/_headers` File

The current `public/_headers` file sets `X-Frame-Options: DENY`, `X-Content-Type-Options: nosniff`, `Referrer-Policy`, `Permissions-Policy`, and `X-XSS-Protection` under `/*`. Since this only applies to static assets (JS, CSS, fonts, images), not HTML, these rules are:

- **Useful for:** Preventing clickjacking framing of SVG/image assets, MIME-type sniffing of JS files
- **Not useful for:** Protecting HTML pages (the actual pages users browse)

The `Cache-Control` rules for `/_next/static/*`, `/images/*`, etc. are correct and working.

**No change needed to `public/_headers`** — its role should just be understood as applying to static file assets only.

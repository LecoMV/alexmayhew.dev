# Accessibility Audit — alexmayhew.dev (2026-04-16)

**Auditor:** accessibility-auditor agent
**Standard:** WCAG 2.1 AA
**Scope:** All routes in `src/app/`, all components in `src/components/`
**Result:** 12 CRITICAL, 19 HIGH, 18 MEDIUM, 8 LOW

---

## Executive Summary

The site has meaningful a11y infrastructure (skip link, `reducedMotion="user"`, global `:focus-visible` outline, semantic `<main>` landmark, lang attribute, keyboard Escape handlers). However, two systemic issues dominate the failure set:

1. **Heading-hierarchy inversion is widespread** — the design system uses a small "eyebrow" label above the visual page title and in many templates the eyebrow is coded as `<h1>` while the actual title is `<h2>`. Ten route templates are affected (services, technologies, for, for/[role], services/[slug], demo, demo/backgrounds, newsletter, blog/list). This fails SC 1.3.1 (Info & Relationships) and 2.4.6 (Headings & Labels) and inverts the document outline for every user of every one of these pages. Screen reader users will hear `●  Technical_Advisory` announced as "heading level 1" with the actual page title demoted to level 2.
2. **Heading-level skips from h1 → h3** — work, tools, blog-list, about, home all jump from the page h1 straight to h3 "subsection" cards, skipping h2 entirely. Fails SC 1.3.1.

Beyond those: the global `:focus-visible` 2px outline in `globals.css:304` does fire on every keyboard-focused interactive element, so the previously-flagged "40+ buttons missing focus ring" finding is partially mitigated in practice — however, many interactive elements still lack explicit Tailwind `focus-visible:ring-*` classes, which means in contexts where the global rule is overridden (custom outline, skin-theme blog layouts with inline `style=`) the focus ring is lost. The global rule is also not what the design system spec requests (`focus:ring-2 focus:ring-cyber-lime focus:outline-none`).

Color contrast on the primary palette clears WCAG AA easily. The contrast failures are concentrated in two patterns: (a) `/50` opacity on `text-slate-text` placeholders (under 3:1), and (b) the `text-white/30–40`, `opacity-40`, `opacity-30` hero decorative/code-listing text which is near-invisible and fails contrast if relied on for content.

## CRITICAL (Block WCAG 2.1 AA)

### C1 — h1 used for eyebrow label, h2 for actual page title (SC 1.3.1, 2.4.6)

Eight page templates code the small decorative label as the page `<h1>` and the visual page title as `<h2>`. This inverts the document outline and breaks SR-user expectation that the h1 conveys the page topic.

- `src/app/services/services-page-content.tsx:114-122` — `h1 = "Technical_Advisory"` (12px), `h2 = "Architecture decisions that…"` (60px)
- `src/app/services/[slug]/service-page-content.tsx:181-194` — `h1 = "{tech} + {industry}"` (eyebrow), `h2 = "{tech} Developer for {industry}"`
- `src/app/technologies/[slug]/technology-page-content.tsx:113-123` — `h1 = "Technology Expertise"`, `h2 = "{tech} Development."`
- `src/app/technologies/technologies-page-content.tsx:52-62` — `h1 = "Technology Expertise"`, `h2 = "Deep expertise in modern technologies."`
- `src/app/for/for-hub-page.tsx:53-64` — `h1 = "Technical Advisory"` (eyebrow), `h2 = "Built for Your Role"`
- `src/app/for/[role]/role-page-content.tsx:105-116` — `h1 = "For {roleLabel}"` (eyebrow), `h2 = "{page.headline}"`
- `src/app/newsletter/page.tsx:55-65` — `h1 = "Newsletter Archive"` (eyebrow), `h2 = "The Architect's Brief."`
- `src/app/demo/page.tsx:94-102` and `src/app/demo/backgrounds/page.tsx:180-188` — same pattern
- `src/app/services/[slug]/not-found.tsx:8` — `h1` set to the eyebrow label
- **Fix:** demote the eyebrow to `<p>` or `<span>` with `role="doc-subtitle"` (or just styled `<p>`) and promote the visual title to `<h1>`.

### C2 — Heading-level skip h1 → h3 (SC 1.3.1, 2.4.6)

Seven files jump from the page `<h1>` straight to `<h3>` for subsection cards, skipping `<h2>`:

- `src/components/pages/work-page.tsx:32, 107` — h1 "Digital Instruments" → project card h3 titles
- `src/components/pages/tools-page.tsx:93, 127` — h1 "Precision Instruments" → tool card h3
- `src/components/pages/about-page.tsx:133, 195` — h1 → "Outcomes Delivered" h3 inside sidebar card (h2 appears later at 223, out of order)
- `src/components/blog/blog-list.tsx:91, 107` — h1 "Engineering & Architecture" → h3 "Comprehensive Guides" (no h2 between)
- `src/app/home-page.tsx:80` — h1 "Strategic Architecture" → Services section has h2 OK, but **Featured Insights** uses h3 cards without a preceding h2 wrapping them inside the section (the section intro is h2 at :303, then cards at :332 use h3 — OK on recount, so home is actually compliant; only issue is that the hero `<m.section>` has no accessible name).
- `src/components/pages/case-study-page.tsx:149` — h3 "Project Details" appears before the first h2 of Key Results at :220 — actually the document has h1 at :64, then h3 at :149 is inside aside/details card before h2. Skip.
- **Fix:** insert an `<h2>` section heading (visible or visually-hidden) before each group of h3 cards, OR demote the cards to h2.

### C3 — Navigation interactive elements have no explicit focus-visible ring (SC 2.4.7)

`src/components/ui/navigation.tsx` lines 126–307 contain 12 `<Link>` and `<button>` elements (logo, all 6 desktop nav links, tools dropdown button, CTA "Book a Call", search trigger, mobile menu button) — none have any `focus-visible:ring-*` class. They rely only on the global `:focus-visible { outline: 2px solid … }` in `globals.css:304-308`. This works in most browsers but (a) fails to meet the design-system contract "`focus:ring-2 focus:ring-cyber-lime focus:outline-none`" documented in `.claude/rules/design-system.md`, and (b) is lost when Tailwind `focus:outline-none` is applied elsewhere (it is not here, but this is fragile). The entire main navigation is therefore a single-rule-removal away from losing focus indication.

- **Fix:** add `focus-visible:ring-2 focus-visible:ring-cyber-lime focus-visible:outline-none` to every interactive element in `navigation.tsx`.

### C4 — Footer links have no explicit focus-visible ring (SC 2.4.7)

`src/components/ui/footer.tsx` lines 64-163: all 9 footer navigation links, 5 social-media icon links, privacy and terms links — none have `focus-visible:ring-*` classes. Same fragility as C3.

### C5 — Upload dropzone not keyboard accessible (SC 2.1.1 Keyboard, 4.1.2 Name/Role/Value)

`src/components/traceforge/upload-zone.tsx:128-190` — the drop/click zone is a `<m.div>` with `onClick={handleClick}` but no `role="button"`, no `tabIndex={0}`, and no `onKeyDown` handler to trigger upload on Enter/Space. A keyboard-only user cannot upload a file on `/tools/traceforge`. The error message at line 196-206 also lacks `role="alert"` so screen-reader users will not hear it.

- **Fix:** either make it a real `<button type="button">`, or add `role="button" tabIndex={0}` plus `onKeyDown={(e) => { if (e.key === 'Enter' || e.key === ' ') { e.preventDefault(); handleClick(); } }}`, and add `role="alert"` to the error block.

### C6 — Chat widget input has no accessible name (SC 4.1.2, 3.3.2 Labels)

`src/components/chat/chat-widget.tsx:258-267` — the chat input has only a `placeholder="Ask me anything..."`. Placeholders are not accessible names. Also no `<label>`, no `aria-label`, no `aria-labelledby`. Screen-reader users hear "edit text" with no context.

- **Fix:** add `aria-label="Chat message"` (or a visually-hidden `<label htmlFor>`).

### C7 — Chat widget message feed is not announced to screen readers (SC 4.1.3 Status Messages)

`src/components/chat/chat-widget.tsx:193-253` — incoming assistant replies are added to the feed without `aria-live="polite"` on the container and without `role="log"`. A screen-reader user who has focused the input and sends a message gets no announcement when the reply arrives.

- **Fix:** wrap the message list in `<div role="log" aria-live="polite" aria-atomic="false">`.

### C8 — Chat widget has no Escape-to-close (SC 2.1.2 No Keyboard Trap — partial; 1.4.13 mitigation)

`chat-widget.tsx:26-116` — Escape does not close the dialog. Keyboard users who open the chat have no keyboard way back to the toggle button except Shift+Tab out.

- **Fix:** add a `useEffect` with `document.addEventListener('keydown', …)` that calls `setIsOpen(false)` on Escape when `isOpen`, mirroring the pattern already used in `navigation.tsx:91-100`.

### C9 — Cookie consent banner lacks dialog semantics and focus management (SC 4.1.2, 2.4.3 Focus Order)

`src/components/ui/cookie-consent.tsx:100-150` — the modal-style banner has no `role="dialog"` / `aria-modal`, no `aria-labelledby` pointing to the Cookie Notice heading, and no initial-focus management (focus remains wherever it was when the banner appeared, so a keyboard user has to Tab through the entire page to reach Accept/Decline). The three buttons (Accept, Decline, Close) also have no `focus-visible:ring-*` classes, so they fall back to the global outline only.

- **Fix:** add `role="dialog" aria-modal="true" aria-labelledby="cookie-notice-heading"`, give the `<h3>` that id, focus the Accept button on mount (useEffect), and add explicit focus rings.

### C10 — Command palette dialog missing accessible name (SC 4.1.2)

`src/components/ui/command-palette.tsx:85-93` — the dialog has `role="dialog" aria-modal="true"` but no `aria-label` and no `aria-labelledby`. Screen-readers announce "dialog" with no context about what it contains.

- **Fix:** add `aria-label="Site search"` to the `<m.div role="dialog">`.

### C11 — Chat widget dialog lacks focus trap (SC 2.4.3 Focus Order)

`chat-widget.tsx:167-281` — open dialog with `role="dialog" aria-label="Chat with AI assistant"` but no focus trap. Tab out of the send button exits the dialog while it is still visually open. Combined with C8, a keyboard user who is inside the chat can escape to the page behind the modal while the modal is still covering content.

- **Fix:** implement a focus trap (or use a library) — trap Tab/Shift-Tab within the dialog until closed.

### C12 — Placeholder contrast fails WCAG AA (SC 1.4.3 Contrast)

Multiple form inputs use `placeholder:text-white/30` (Contact page inputs at `contact-page.tsx:169, 185, 317`; Chat at `chat-widget.tsx:266`; GPU-control at `traceforge/gpu-control.tsx:133` uses `placeholder:text-slate-text/50`). On `bg-void-navy` (#0B0E14), `text-white/30` resolves to approximately #4D4D4D which against #0B0E14 yields roughly **2.6:1** — below the 4.5:1 required for AA normal text. Placeholder text is considered essential in this design because labels are in small caps above.

- **Fix:** raise to `placeholder:text-slate-text` (#94A3B8 on #0B0E14 ≈ 7.0:1) or `placeholder:text-white/60`.

## HIGH (Meaningfully degrades assistive tech UX)

### H1 — Work page filter tabs have no selected-state semantics (SC 4.1.2)

`src/components/pages/work-page.tsx:50-70` — filter buttons change the grid but set no `aria-pressed` or `aria-selected`. Screen-reader users cannot tell which filter is active; they only hear "button".

- **Fix:** add `aria-pressed={activeCategory === category}` on each button, and wrap with `role="group" aria-label="Filter projects by category"`.

### H2 — Newsletter-signup "minimal" variant has no error surface (SC 3.3.1 Error Identification)

`src/components/newsletter/newsletter-signup.tsx:205-235` — the minimal/footer form omits the `{state.error && …}` block entirely. Invalid emails submitted from the footer yield no visible or announced error; the user sees nothing.

- **Fix:** render the same `role="alert"` error block as the other variants.

### H3 — Chat widget dialog has no visible close button inside the dialog (SC 2.4.3)

The only close is the toggle button floating outside the dialog at bottom-right. Inside the dialog there is no `X` button. Combined with C8 (no Escape), keyboard users navigate via the external toggle — fragile.

- **Fix:** add a close `<button aria-label="Close chat">` inside the dialog header.

### H4 — Contact form missing `<fieldset>/<legend>` for grouped selects (SC 1.3.1)

`src/components/pages/contact-page.tsx:156-302` — Name/Email row and Project Type/Budget row are grouped visually with `.grid md:grid-cols-2` but have no `<fieldset>/<legend>`. Not required but recommended for longer grouped forms, and the page-level form has no `<form aria-labelledby>` either. At minimum, `<form aria-label="Contact form">` should be added.

### H5 — Blog table-of-contents buttons have no focus ring and mobile toggle has no aria-expanded (SC 2.4.7, 4.1.2)

`src/components/blog/table-of-contents.tsx:349, 394, 446, 474, 503` — all TOC buttons (both desktop and mobile) have no `focus-visible:ring-*` class. The mobile toggle at :446 has no `aria-expanded` / `aria-controls` for the expandable list.

- **Fix:** add focus rings and `aria-expanded={open}` + `aria-controls="toc-list"` on the mobile toggle.

### H6 — Command palette dialog lacks Backdrop button semantics (SC 2.1.1)

`command-palette.tsx:79-82` — backdrop `<div>` is clickable to close (`onClick={() => setOpen(false)}`) but has no role/tabIndex/keyboard handler. Acceptable because Escape closes via cmdk — but this relies on cmdk's behavior. Confirm cmdk handles Escape (it does by default).

### H7 — Share buttons: "COPIED!" feedback not announced (SC 4.1.3)

`src/components/blog/share-buttons.tsx:210-221` — the "Link copied to clipboard" message appears as a static `<m.div>`. Screen-reader users hear nothing after pressing the Copy button.

- **Fix:** add `role="status" aria-live="polite"` to the feedback `<m.div>`.

### H8 — Ellipsis/decorative pulse bullets announced as "black circle"/"bullet" without aria-hidden (SC 1.3.1)

The `<span className="mr-2 animate-pulse">●</span>` decoration appears before many headings (home, about, services, blog, contact, newsletter, tools, work, footer, all pSEO pages). Most use `aria-hidden="true"` (navigation:175, contact:130, footer:58, home:77). However many do NOT:

- `src/app/home-page.tsx:185, 248, 304, 362, 392` — hero/services/about/featured/newsletter/CTA eyebrow bullets
- `src/components/pages/tools-page.tsx:90` — "Developer Tools" eyebrow
- `src/app/services/services-page-content.tsx:115`
- `src/app/technologies/technologies-page-content.tsx:53`
- `src/app/for/for-hub-page.tsx:54`
- `src/components/pages/work-page.tsx:29`
- `src/components/pages/about-page.tsx:130, 196`
- plus many pSEO page files
  Screen readers pronounce `●` (U+25CF) as "black circle". Every heading prefix is announced with a meaningless "black circle" before the real label.
- **Fix:** add `aria-hidden="true"` to every decorative `●` / `▸` / `◆` span.

### H9 — Tools page grid has no `aria-labelledby` for sections (SC 1.3.1)

`src/components/pages/tools-page.tsx:105+` — the grid wraps `<m.article>` but the page lacks `<section aria-labelledby>` to group the tool cards. Affects screen-reader landmark navigation.

### H10 — Chat widget notification-dot has no accessible indication (SC 1.3.3 Sensory Characteristics)

`chat-widget.tsx:159-161` — the pulsing green dot that signals "new message / unread" is visual-only. No equivalent text or `aria-label` update on the toggle button.

- **Fix:** include "(new)" in the `aria-label` when the dot is showing.

### H11 — Progress bar lacks aria-valuenow (SC 4.1.2)

`src/components/blog/reading-progress.tsx:3-9` — `role="progressbar"` present but no `aria-valuenow` / `aria-valuemin` / `aria-valuemax`. Because the actual progress calculation is CSS-only (scroll-driven), screen readers cannot report progress.

- **Fix:** remove `role="progressbar"` (mark decorative with `aria-hidden="true"`) OR calculate progress in JS and set `aria-valuenow`.

### H12 — Focus ring globally set to 2px `outline` but design system specifies ring+offset (SC 2.4.7 adequacy)

`src/app/globals.css:304-308` — `:focus-visible { outline: 2px solid var(--color-cyber-lime); outline-offset: 2px; }`. The outline-offset is good, but combined with dark backgrounds the 2px cyan-lime outline has 16:1 contrast against void-navy — passes AA for non-text contrast. However, inside cards with `bg-gunmetal-glass` (lighter) the outline is thinner against the similar-luminance surface; the `outline-offset` helps. This is adequate but fragile: any component that sets `outline: none` (common in custom focus handling) silently loses the ring. Only 6 of the 55+ files that contain interactive elements use `focus-visible:ring-*` classes.

- **Fix:** roll out `focus-visible:ring-2 focus-visible:ring-cyber-lime focus-visible:ring-offset-2 focus-visible:ring-offset-void-navy` to every interactive element, or move the rule into a Tailwind component layer so it cannot be overridden locally.

### H13 — Heading hierarchy in migration/integration/comparison pages uses `<m.h1>` (OK) but sub-sections go h2 → h4 skipping h3 (SC 1.3.1)

`src/app/services/integrations/[slug]/integration-page-content.tsx:480, 492, 564, 575, 619, 623, 627, 741, 805` — multiple `<h4>` inside `<h2>` sections with no intervening `<h3>`. Same pattern in comparison and migration pages.

- **Fix:** demote h4 → h3 or add intermediate h3 subsection headings.

### H14 — Contact form success message not announced (SC 4.1.3)

`src/components/pages/contact-page.tsx:352-360` — the "Message received" success `<m.p>` has no `role="status"` / `aria-live`. Users who submit the form via keyboard won't hear confirmation. (The error path at :363-373 correctly uses `role="alert"`.)

### H15 — GPU-control error message not announced (SC 4.1.3)

`src/components/traceforge/gpu-control.tsx:160-168` — error `<m.p>` has no `role="alert"`.

### H16 — GPU password input has no visible label (SC 3.3.2 Labels)

`traceforge/gpu-control.tsx:124-141` — password `<input>` has only `placeholder="Admin password..."`, no `<label>`, no `aria-label`. Password managers may also struggle.

- **Fix:** add `aria-label="Admin password"`.

### H17 — Skip link text mismatches landmark id naming convention — minor (SC 2.4.1)

`src/components/ui/navigation.tsx:120-125` — "Skip to content" targets `#main-content`. The landmark `<main id="main-content">` is correctly set in `layout.tsx:162`. Works. Low concern but the link uses plain `<a>`, not `<Link>` — good (avoids client routing for a same-page anchor). Verify that clicking the skip link moves focus as well as scroll — current implementation relies on browser default which does not always move focus reliably in SPAs. Consider `onClick` handler to call `.focus({ preventScroll: false })` on the main element (and add `tabIndex={-1}` to `<main>`).

### H18 — Cookie consent Accept/Decline buttons lack explicit focus rings (SC 2.4.7)

`src/components/ui/cookie-consent.tsx:124-144` — three buttons, none have `focus-visible:ring-*` classes.

### H19 — Tools page h1 followed directly by h3 tool cards (SC 1.3.1, covered under C2)

See C2.

## MEDIUM (Notable but does not block compliance)

### M1 — Hero section on home page has `<m.section>` with no accessible name (SC 1.3.6 Identify Purpose, AAA; SC 2.4.1 informational)

`src/app/home-page.tsx:69-125` — the hero is `<m.section>` but has no `aria-label` or heading-reference. Low severity because the h1 inside gives context, but a `aria-labelledby` would improve landmark navigation.

### M2 — MDX image default alt = empty string (SC 1.1.1)

`src/components/mdx/mdx-components.tsx:147` — `alt={alt ?? ""}` — sets alt to empty if author forgets. If the image is informational this is a silent 1.1.1 violation. Consider failing the build or warning when an MDX image lacks alt.

### M3 — Redundant alt text on blog featured image (SC 1.1.1 advisory)

`src/components/blog/blog-article.tsx:179` and `src/components/blog/blog-list.tsx:137`, `cards-layout.tsx:103, 245, 351`, `dossier-layout.tsx:86` — all use `alt={post.data.title}`. The title is immediately adjacent as `<h1>` / `<h3>`, so screen readers announce the title twice. Better: empty alt (decorative) since the heading conveys the same info, or use a genuinely descriptive alt about the image content.

### M4 — AM logo alt = "AM" in nav and footer (SC 1.1.1)

`navigation.tsx:139`, `footer.tsx:67` — alt "AM" is technically valid but not descriptive. The wordmark next to it already says "alexmayhew.dev". Consider `alt=""` (decorative) since the text wordmark conveys the brand.

### M5 — Breadcrumb pattern uses `<m.nav>` without `<ol>` in service pages (SC 1.3.1)

`src/app/services/services-page-content.tsx:95-105` — breadcrumb is `<m.nav>` but uses inline `<Link>` + `<span>` separators rather than a `<ol>`. For-hub page and migrations pages correctly use `<ol>` (for-hub-page.tsx:37). Inconsistency.

### M6 — 10px font size widely used on mobile (SC 1.4.4 Resize Text)

23 files use `text-[10px]`, `text-[9px]`, `text-[8px]` (command-palette, badges, terminal, pSEO badges, etc.). WCAG 2.1 does not mandate a minimum size but requires that content remains usable at 200% zoom. 8px text at 200% zoom is 16px, marginally acceptable. The 9px blog layout and 10px heading-prefix code labels (`navigation.tsx: 168, 263`) will be difficult for low-vision users even at 200%. Consider a floor of 11px for all body content.

### M7 — `text-white/30`, `text-white/40`, `opacity-30–50` decorative code blocks fail contrast if read (SC 1.4.3)

`src/app/home-page.tsx:136-158` — hero code-listing has `text-white/40`, `text-white/20`, `opacity-40/30` — contrast ~1.8:1. Acceptable because the content is decorative (stylized code). Ensure screen readers don't read it (the ` const philosophy = { … }` block is not marked `aria-hidden`).

- **Fix:** wrap in `<div aria-hidden="true">`.

### M8 — Terminal component input has aria-label but output feed not live (SC 4.1.3)

`src/components/terminal/terminal.tsx:198-235` — terminal output scrolls but `<div>` wrapper has no `role="log"` or `aria-live="polite"`. Users of the terminal who type commands don't hear the responses.

- **Fix:** `role="log" aria-live="polite" aria-atomic="false"` on output scroll container.

### M9 — Heading size vs weight: visual h3 is larger than h2 (SC 1.3.1 semantics-vs-visuals)

On services/about/technologies pages, the eyebrow h1 (inverted — see C1) is rendered at 12px and h2 at 60px. Once C1 is fixed by demoting to `<p>`, the remaining h1 = 60px / h2 = ~32px / h3 = 16-20px — correct ordering.

### M10 — `<m.nav>` wrapped animated breadcrumb with `aria-label="Breadcrumb"` but Link inside has no aria-current="page" (SC 2.4.8)

Some breadcrumb templates (for-hub-page.tsx:44) correctly use `aria-current="page"`. Others (services-page-content.tsx:104) use styling only. Make consistent.

### M11 — No `aria-current="page"` on active nav link (SC 2.4.8)

`navigation.tsx:256-274` — active nav items get a visual underline (`isActive && …`) but no `aria-current="page"`. Screen readers don't know which page the user is on.

- **Fix:** add `aria-current={isActive ? "page" : undefined}` to each nav `<Link>`.

### M12 — Mobile menu overlay doesn't trap focus (SC 2.4.3)

`navigation.tsx:311-436` — mobile menu opens below the menu button. Focus isn't trapped inside the opened menu. On small screens, this means Tab can escape out of the open menu onto page content that's visually covered. Moderate severity — Escape does close (line 91-100).

### M13 — Footer social icons lack focus ring classes (SC 2.4.7)

`footer.tsx:115-131` — social `<a>` elements have `hover:border-cyber-lime` but no `focus-visible:ring-*`.

### M14 — Typography sm text color contrast edge cases

`text-slate-text/60` and `text-slate-text/50` on some pages drops contrast to ~4.0:1 on void-navy — borderline for 4.5:1 normal AA. Acceptable for large (≥18px/24px bold) but verify.

### M15 — Form labels are visually all-caps but HTML uses lowercase text (SC 3.3.2 advisory)

`contact-page.tsx:159, 175, 194, etc.` — labels use Tailwind `uppercase` CSS. The underlying text is "Name", "Email", etc. — fine for a11y. Not a violation; noting as a style observation.

### M16 — Turnstile widget fallback (dev only) has no role/label (SC 4.1.2)

`src/components/ui/turnstile.tsx:32-34` — the `[Turnstile: Set NEXT_PUBLIC_TURNSTILE_SITE_KEY]` placeholder is dev-only but has no context. Low priority (dev-only).

### M17 — Scrollable content areas lack `tabindex=0` (SC 2.1.1)

Long scrollable regions (chat feed, TOC, command-palette list) don't have `tabindex="0"` on their scroll containers. Keyboard-only users who can't scroll via a scrollbar need a way to keyboard-scroll these. WCAG 2.1.1 technically only requires _operable_ elements be keyboard-accessible, and scrolling is typically achieved via focus within — acceptable.

### M18 — Chat send button target size (SC 2.5.8 in 2.2 — AA in 2.2 only, AAA in 2.1)

`chat-widget.tsx:268-275` — `h-10 w-10` = 40×40px. Below 44×44 but above 24×24 AA minimum. In WCAG 2.2 this is a Level AA failure; in 2.1 it's AAA. Note: the scope says WCAG 2.1, so this is AAA-level.
